## 1. Что это за система

AmreGroup ERP — это внутренняя веб-система управления финансами компании. Она заменяет Excel-таблицы и разрозненные выгрузки из банков и 1С.

**Главная задача системы:** дать руководству компании полную картину финансового состояния бизнеса в режиме реального времени — сколько денег на счетах, откуда они приходят, куда уходят, какой план на месяц и как он выполняется.

**Кто пользуется:**
- Руководитель / Учредитель — видит дашборд, отчёты, баланс
- Финансовый директор / Бухгалтер — вводит транзакции, делает сверки, импортирует банковские выписки
- Менеджеры проектов — видят расходы по своим проектам

## 2. Технологический стек

| Категория | Технология |

| Авторизация | Firebase Authentication |

**Почему Firebase:** данные обновляются в реальном времени без перезагрузки страницы. Открыл отчёт — цифры живые, без кнопки "Обновить".

## 3. Архитектура системы

```
src/
├── pages/          — Страницы (что видит пользователь)
│   ├── Finance/    — Все финансовые страницы (18 страниц)
│   └── Directories/— Справочники
├── services/       — Работа с базой данных (чтение/запись)
├── hooks/          — Логика состояния (расчёты, подписки на данные)
├── models/         — Структуры данных (что хранится в БД)
├── components/     — Переиспользуемые UI-компоненты
└── utils/          — Вспомогательные функции (форматирование, экспорт)
```

**Принцип работы:**
1. Данные живут в Firebase Firestore (облако)
2. `services/` — пишут и читают данные из Firestore
3. `hooks/` — подписываются на данные и считают производные значения (например, текущий баланс счёта)
4. `pages/` — показывают данные пользователю и принимают действия

## 4. Модули системы

### 4.1 Финансовый модуль (основной, полностью реализован)

Это ядро системы. Состоит из 4 блоков:

---

#### Блок A: Операции

**Журнал транзакций** (`/finance/transactions`)  
Список всех финансовых операций компании. Каждая транзакция имеет:
- Тип: **Доход / Расход / Перевод** между счетами
- Статус: **Факт** (уже прошло) или **План** (ожидается)
- Сумма, дата платежа, дата начисления (могут отличаться)
- Счёт (с какого/на какой банк/касса)
- Категория (статья ДДС или ОПиУ)
- Контрагент
- Проект
- Теги (для дополнительной фильтрации)
- Описание, хеш дедупликации

Возможности: поиск, фильтры по любому полю, inline-редактирование, массовые операции, добавление через модал.

---

**Платёжный календарь** (`/finance/calendar`)  
Визуальный календарь плановых платежей. Показывает что нужно оплатить в ближайшие дни. Функции:
- Drag & Drop — перетащить платёж на другой день
- Повторяющиеся платежи (аренда, зарплата, SubscriptionService)
- График ликвидности — хватит ли денег с учётом плановых платежей
- Экспорт в Excel

---

**Планирование** (`/finance/planning`)  
Бюджет на текущий месяц. Менеджер заполняет сколько планирует потратить по каждой категории (зарплата, материалы, подрядчики, аренда и т.д.). Это основа для отчёта План-факт.

---

#### Блок B: Импорт и сверка

**Импорт банковской выписки** (`/finance/import`)  
Загрузка выписки из банка напрямую — система сама разбирает файл.

Поддерживаемые форматы:
- Halyk Bank
- Kaspi Bank
- 1С (стандартный формат)

Процесс импорта:
1. Загрузи файл
2. Система находит похожие транзакции (дедупликация по MD5-хешу)
3. Применяются **авто-правила** — система автоматически расставляет категории по паттернам
4. Предпросмотр: можно поправить категорию/проект для каждой строки вручную
5. Сохранение в базу (до 500 операций за раз)

---

**Авто-правила** (`/finance/auto-rules`)  
Настраиваемые правила для автоматической категоризации при импорте.

Пример правила: *"если в описании есть слово 'аренда' и сумма от 500 000 до 1 000 000 тенге → категория 'Аренда офиса', проект 'Операционные расходы'"*

Три уровня правил (применяются по очереди):
1. Пользовательские правила (созданные в интерфейсе)
2. Системные правила (захардкоженные паттерны: зарплата, налоги, материалы)
3. AI-категоризация (если ничего не подошло)

---

**Сверка** (`/finance/reconcile`)  
Сравнение трёх источников данных: банковский файл vs 1С vs база системы.

Показывает три группы:
- Есть в файле, но не в базе (пропущено при импорте)
- Есть в базе, но не в файле (лишнее в базе)
- Совпадает везде

Помогает убедиться что данные полные и корректные.

---

#### Блок C: Аналитика и отчёты

**Дашборд** (`/finance`)  
Главная страница. Сводные метрики за выбранный год:
- Общий доход / расход / прибыль
- График cash flow по месяцам (столбчатая диаграмма)
- Фильтры: год, месяц, проект

---

**ДДС — Движение Денежных Средств** (`/finance/analytics/cashflow`)  
Классический отчёт ДДС. Таблица: строки — статьи, колонки — месяцы.

Структура ДДС:
- **Операционная деятельность** — основной бизнес (выручка, зарплата, аренда, материалы)
- **Инвестиционная деятельность** — оборудование, стройка, покупка активов
- **Финансовая деятельность** — займы, дивиденды, взносы учредителей

Экспорт в Excel.

---

**ОПиУ — Отчёт о Прибылях и Убытках** (`/finance/analytics/pnl`)  
Управленческий P&L. Структура:
```
Выручка (Revenue)
- Себестоимость (COGS)
= Валовая прибыль (Gross Profit)
- Операционные расходы (OPEX)
= EBITDA
- Амортизация
= Чистая прибыль (Net Profit)
```

Два метода учёта:
- **Кассовый** — когда деньги пришли/ушли
- **Начисления** — когда услуга оказана (дата начисления ≠ дата платежа)

---

**Управленческий баланс** (`/finance/analytics/balance`)  
Баланс компании на любую дату. Структура:

```
АКТИВЫ                          ПАССИВЫ
Деньги на счетах                Займы (кредиторы)
Дебиторская задолженность       Кредиторская задолженность
Товарные запасы                 Уставной капитал
Внеоборотные активы             Добавочный капитал
                                Нераспределённая прибыль
```

Денежные остатки считаются автоматически из транзакций. Запасы, внеоборотные активы и статьи капитала вводятся вручную.

---

**План-факт** (`/finance/analytics/planfact`)  
Сравнение бюджета с реальными расходами по каждой категории.

Колонки: Категория | План | Факт | Отклонение (₸) | Отклонение (%)

Бюджетные лимиты редактируются прямо в таблице inline.

---

**Отчёт по проектам** (`/finance/analytics/projects`)  
Финансовый результат по каждому проекту:
- Выручка
- Себестоимость
- OPEX
- Прибыль
- Маржа (%)

---

**Дебиторка / Кредиторка** (`/finance/analytics/debts`)  
Отчёт по долгам контрагентов. Начислено по 1С vs оплачено по банку → остаток долга по каждому партнёру.

---

**Выписка по счёту** (`/finance/analytics/account-statement`)  
Движение по выбранному счёту: дата, описание, приход, расход, бегущий остаток. Как банковская выписка, но внутри системы.

---

**Взаиморасчёты** (`/finance/settlements`)  
Сальдо по каждому контрагенту с разбивкой по проектам. Кто нам должен, кому должны мы.

---

**Зарплата** (`/finance/payroll`)  
Расчёт ФОТ за месяц. Загружается банковская выписка → система автоматически сопоставляет платежи с сотрудниками → статусы выплат.

---

**Сводный финансовый отчёт** (`/finance/analytics/financials`)  
Revenue, COGS, OPEX по месяцам в одной таблице. Быстрый обзор для руководства.

---

**Аналитический хаб** (`/finance/analytics`)  
Страница-навигатор: карточки со ссылками на все аналитические отчёты.

---

#### Блок D: PWA-страницы (для мобильных сценариев)

**Подотчёт учредителя** (`/founder/:token`)  
Закрытая страница, доступная по уникальному токену или PIN-коду. Учредитель вносит личные расходы связанные с бизнесом прямо с телефона, без входа в основную систему.

## 6. Ключевые технические особенности

### Дедупликация при импорте
Каждая транзакция при импорте получает MD5-хеш из полей (дата + сумма + описание + счёт). Перед сохранением система проверяет — нет ли уже транзакции с таким хешем. Это защищает от случайного двойного импорта одной выписки.

### Realtime балансы
Балансы счетов в шапке страницы обновляются в реальном времени через Firestore `onSnapshot`. Добавил транзакцию → баланс изменился мгновенно без перезагрузки.

### Дата платежа vs Дата начисления
Это ключевое разделение для управленческого учёта. Например: оплатили поставщику 1 марта, но услуга была оказана в феврале. В ДДС это уйдёт в март (когда деньги ушли), а в ОПиУ в февраль (когда услуга оказана). Система хранит обе даты.

### Блокировка периода
Финансовый директор может "закрыть" период (например январь). После этого транзакции за январь нельзя редактировать — это защита от случайного изменения уже отчитанных данных.

### Пакетный импорт
Firestore имеет ограничение на одну операцию записи. Система пишет данные пакетами по 500 транзакций за раз (batch commit) для надёжности.

## 7. Что нужно изменить для FixPlast ERP

Проект `fixplast-erp` — это копия данной системы, адаптированная под компанию FixPlast. Финансовый движок (ДДС, ОПиУ, баланс, импорт) остаётся тот же. Нужно изменить следующее:

### 7.1 Обязательные изменения

| Что менять | Где | Описание |
|---|---|---|
| Firebase проект | `.env` / `firebase.json` | Создать новый Firebase проект для FixPlast, подключить новую БД |
| Название компании | `src/App.tsx`, `Sidebar.tsx`, заголовки | Поменять "AmreGroup" → "FixPlast" |
| Логотип / брендинг | `public/`, `Sidebar.tsx` | Цвета, логотип компании |
| Справочник категорий | Firebase консоль | Свои статьи доходов/расходов под бизнес FixPlast |
| Список счетов | Firebase консоль | Банковские счета FixPlast |
| Firebase Authentication | Firebase консоль | Создать пользователей для сотрудников FixPlast |
| Правила Firestore | `firestore.rules` | Проверить/адаптировать права доступа |

### 7.2 Возможные бизнес-отличия (уточнить с заказчиком)

| Вопрос | Влияние на систему |
|---|---|

## 10. Глоссарий

| Термин | Расшифровка |
|---|---|
| ДДС | Движение Денежных Средств (Cash Flow Statement) |
| ОПиУ | Отчёт о Прибылях и Убытках (P&L / Income Statement) |
| Факт | Транзакция которая уже произошла |
| План | Ожидаемая транзакция (будущий платёж) |
| Дата начисления | Дата когда услуга/товар получены (для метода начисления) |
| Дата платежа | Дата когда деньги реально прошли по счёту |
| Контрагент / Партнёр | Клиент, поставщик или подрядчик |
| COGS | Cost of Goods Sold — себестоимость реализованной продукции |
| OPEX | Operating Expenses — операционные расходы |
| EBITDA | Прибыль до вычета налогов, процентов и амортизации |
| PWA | Progressive Web App — сайт, который устанавливается как мобильное приложение |
| Firebase | Облачная платформа Google (база данных + хостинг + авторизация) |
| Firestore | NoSQL база данных от Firebase с realtime-обновлениями |
| Batch | Пакетная запись в БД (до 500 операций за раз) |
| MD5-хеш | Цифровой отпечаток транзакции для защиты от дублей при импорте |
